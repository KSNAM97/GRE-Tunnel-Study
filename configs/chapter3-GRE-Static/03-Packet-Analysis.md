# Chapter 3-3. 패킷 캡슐화 분석 (Dynagen)

GRE Tunnel을 통해 흐르는 패킷의 캡슐화 구조를 Wireshark로 분석합니다.

## 🎯 Dynagen 캡처 명령어

```bash
=> capture R1 s1/0 GRE_R1.cap FR
```

R1(SOL-A)의 Serial 1/0 인터페이스에서 Frame-Relay 포맷으로 캡처합니다.

---

## 1️⃣ Tunnel IP 간 통신 (SOL-A ↔ SOL-B)

```bash
SOL-A# ping 172.16.10.2
Sending 5, 100-byte ICMP Echos to 172.16.10.2, timeout is 2 seconds:
!!!!!
Success rate is 100 percent (5/5), round-trip min/avg/max = 20/30/56 ms
```

### SOL-A → SOL-B (ICMP Request)

```
==========================================================================
| FR DLCI | New IP            | GRE      | Original IP        | ICMP    |
--------------------------------------------------------------------------
|   102   | SA: 121.160.10.1  | Type     | SA: 172.16.10.1    | Type-8  |
|         | DA: 121.160.20.2  | 0x0800   | DA: 172.16.10.2    | Request |
==========================================================================
```

### SOL-A ← SOL-B (ICMP Reply)

```
==========================================================================
| FR DLCI | New IP            | GRE      | Original IP        | ICMP    |
--------------------------------------------------------------------------
|   102   | SA: 121.160.20.2  | Type     | SA: 172.16.10.2    | Type-0  |
|         | DA: 121.160.10.1  | 0x0800   | DA: 172.16.10.1    | Reply   |
==========================================================================
```

## 2️⃣ 사설망 간 통신 (PC1 ↔ PC2) — Static Route 적용 후

### PC1 (R7) → SOL-A (R1)

```
========================================================
| Ethernet     | IP                 | ICMP            |
| SA: PC1 Mac  | SA: 192.168.10.1   | Type-8 Request  |
| DA: G/W Mac  | DA: 192.168.20.2   |                 |
========================================================
```

### SOL-A → ISP-1 ~ ISP-4 (Frame-Relay + GRE 캡슐화)

```
==========================================================================
| FR DLCI | New IP            | GRE      | Original IP        | ICMP    |
--------------------------------------------------------------------------
|   103   | SA: 121.160.10.1  | Type     | SA: 192.168.10.1   | Type-8  |
|         | DA: 121.160.20.2  | 0x0800   | DA: 192.168.20.2   | Request |
==========================================================================
```

### ISP-4 → SOL-B

```
==========================================================================
| FR DLCI | New IP            | GRE      | Original IP        | ICMP    |
--------------------------------------------------------------------------
|   602   | SA: 121.160.10.1  | Type     | SA: 192.168.10.1   | Type-8  |
|         | DA: 121.160.20.2  | 0x0800   | DA: 192.168.20.2   | Request |
==========================================================================
```

### SOL-B (R2) → PC2 (R8)

```
========================================================
| Ethernet     | IP                 | ICMP            |
| SA: G/W Mac  | SA: 192.168.10.1   | Type-8 Request  |
| DA: PC2 Mac  | DA: 192.168.20.2   |                 |
========================================================
```

---

## 📌 핵심 정리

- **공중망 (ISP)** 에서는 `New IP Header (121.160.x.x)` 로만 라우팅됨
- **원본 사설 IP (192.168.x.x)** 는 GRE Payload 안에 캡슐화되어 숨겨짐
- 양쪽 끝단 (SOL-A, SOL-B)에서만 GRE 캡슐화/디캡슐화가 발생
- 그래서 ISP 라우터는 사설망 정보를 알 필요가 없음
